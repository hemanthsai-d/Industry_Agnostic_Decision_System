Lifecycle

Model Deployment Lifecycle

New models don't go from training straight to production. They go through a careful 5-stage process to make sure they perform well before serving real customers.

🧪 Shadow Mode 0% traffic
→
🔬 Canary 5% 5% traffic
→
📊 Canary 25% 25% traffic
→
⚖️ Canary 50% 50% traffic
→
🚀 Full Production 100% traffic
🧪

Stage 1: Shadow Mode

What happens: The new model runs alongside the current one, but its predictions are only logged, not used. Real customers still get answers from the old model.

Why: We can compare the new model's accuracy against the current one without any risk.

🔬

Stage 2-4: Canary Rollout

What happens: The new model gradually serves 5% → 25% → 50% of real traffic. At each stage, quality gates are checked before increasing.

Why: If the model has issues, only a small percentage of customers are affected, and we catch problems early.

Traffic Splitting

Deterministic Canary Bucketing

How does the system decide which customers get the new model? It uses a mathematical hash to ensure the same customer always gets the same model version — no randomness, fully reproducible.

🎯 SHA-256 Bucketing Logic
# For every request: bucket = SHA-256( tenant_id + ":" + request_id ) mod 100 # If bucket < canary_percent → use new model # Otherwise → use current model # Example with 25% canary: # bucket = 17 → New model (17 < 25) ✓ # bucket = 73 → Current model (73 ≥ 25) ✗

Why this matters: Unlike random splitting, SHA-256 hashing means the same tenant+request always lands in the same bucket. This makes results reproducible and debugging easier. No external feature flag systems needed.

Quality Control

Promotion Quality Gates

Before a model can move to the next stage, it must pass ALL of these gates. If any gate fails, the promotion is blocked and the team is notified.

🎯 Routing Accuracy ● PASS
82.4% Target: ≥ 75%

What this means: When the AI classifies a customer message into one of 30 categories, it gets it right 82.4% of the time. The target is at least 75%.

🚨 Escalation Recall ● PASS
78.6% Target: ≥ 70%

What this means: Of all the cases that truly should be escalated to a human, the system correctly flags 78.6% of them. Missing escalations is dangerous — that's why we track this.

📏 Calibration Error (ECE) ● PASS
0.08 Target: ≤ 0.15

What this means: When the system says "I'm 80% confident," it should be right about 80% of the time. ECE = 0.08 means confidence scores are well-calibrated (honest).

⏸️ Abstain Rate ● PASS
22.1% Target: ≤ 35%

What this means: The system abstains (defers to humans) 22.1% of the time. Too high would mean the AI isn't automating enough; too low would mean it's being reckless.

Daily Operations

Automated ML Operations Pipeline

Every day, this automated pipeline runs to keep the models healthy and improving.

📊 Step 1: Build Daily Evaluation Dataset

Collects all decisions made yesterday, matches them with any human feedback or corrections, and builds a labeled evaluation dataset.

make evaluate-daily

📈 Step 2: Compute Daily Metrics

Calculates routing accuracy, escalation recall, calibration error, and abstain rate from yesterday's data. Updates Prometheus gauges.

make evaluate-metrics

🔍 Step 3: Drift Detection

Compares today's input patterns, confidence distributions, and outcome distributions against baselines. If something has shifted significantly, alerts fire.

make drift-check

🚀 Step 4: Canary Promotion Check

Evaluates if the challenger model meets all quality gates. If it does, automatically advances to the next traffic percentage. If not, the promotion is blocked.

make promote-canary

🔧 Step 5: Recalibration (Weekly)

Once a week, re-fits the Platt scaling calibration using the latest human feedback data. This keeps confidence scores honest as the world changes.

make recalibrate-models

AI Search

Hybrid Retrieval System

When the system needs to find relevant knowledge, it uses TWO different search methods and combines them for the best results. Think of it as using both a dictionary AND Google to find information.

🔤

Full-Text Search (FTS)

How it works: Traditional keyword matching powered by PostgreSQL. Searches for exact words and stemmed variations.

Good at: Finding documents that contain the exact words the customer used.

Limitation: Can't understand synonyms or paraphrases.

🧠

Dense Vector Search

How it works: Converts text into 384-dimensional mathematical vectors using AI (Sentence-Transformers), then finds similar vectors.

Good at: Understanding meaning — "charged twice" finds documents about "duplicate billing."

Limitation: Sometimes too creative, might miss exact matches.

🔀 Combined Retrieval Pipeline
📝 Customer Query
→
🔤 FTS Results
&
🧠 Vector Results
→
🔀 RRF Fusion
→
🏆 Re-Rank

Reciprocal Rank Fusion (RRF) merges both result lists, then cross-encoder reranking selects the best evidence

⚡

Local (Dev)

Hash-based 64-dim vectors. Fast but low quality. For development and CI testing.

EMBEDDING_BACKEND=local Dimension: 64
🧠

Sentence-Transformer

all-MiniLM-L6-v2 model. 384-dim vectors. Production default — great accuracy, runs locally.

EMBEDDING_BACKEND=sentence-transformer Dimension: 384
☁️

Cloud API

OpenAI embeddings. 1536-dim vectors. Highest quality for enterprise deployments.

EMBEDDING_BACKEND=api Dimension: 1536
Safety Net

Drift Detection

The real world changes — customer issues evolve, new products launch, language shifts. Drift detection catches when the AI's performance starts degrading because the world has changed.

📊

Input Drift

Monitors the distribution of incoming messages. If customers start asking about unfamiliar topics, this alerts the team that the model may need retraining.

Alert: DecisionApiInputDriftDetected

📏

Confidence Drift

Tracks if the model's confidence distribution changes significantly. A sudden drop in confidence across the board often signals a real problem.

Alert: DecisionApiConfidenceDriftDetected

📈

Outcome Drift

Watches for changes in the ratio of recommend/abstain/escalate decisions. If escalations suddenly spike, something is wrong.

Alert: DecisionApiOutcomeDriftDetected
